# CLAUDE.md — Next.js 15 + SQLite SaaS Project

## Stack & Versions

- **Framework:** Next.js 15 (App Router)
- **Language:** TypeScript strict mode
- **Database:** SQLite via better-sqlite3 (production) / Turso (edge, optional)
- **ORM:** Drizzle ORM (never use Prisma or TypeORM)
- **Auth:** NextAuth.js v5 / Auth.js
- **Styling:** Tailwind CSS v4 + shadcn/ui
- **Forms:** React Hook Form + Zod
- **Testing:** Vitest + Playwright
- **Package Manager:** pnpm (never npm or yarn)

## Database Rules

### Schema Design
- One Drizzle schema file per domain entity (`users.ts`, `teams.ts`, `subscriptions.ts`)
- Always use `id` as primary key: `id: integer().primaryKey({ autoIncrement: true })`
- Always add `createdAt` and `updatedAt` timestamps
- Use `text()` for strings, `integer()` for booleans (0/1), `real()` for decimals
- Never use `json` column type — use a separate table for one-to-many relations
- Index every foreign key column
- Index columns used in `WHERE` and `ORDER BY` clauses

### Migrations
- Generate: `pnpm drizzle-kit generate`
- Apply: `pnpm drizzle-kit migrate`
- Never edit migration files manually
- For rollback: restore DB from backup, re-run migrations
- Use one migration per schema change (squash frequently)

### Query Patterns
- Write queries in `src/db/queries/`, not inline in components
- Use prepared statements for frequently executed queries
- Never use raw SQL strings — always use Drizzle query builder
- For complex reports: use Drizzle views or raw SQL in a dedicated `reports/` directory

## Auth & Session Pattern

```
src/
├── lib/auth.ts       # Auth.js config (providers, callbacks, adapter)
├── actions/auth.ts   # Login, register, logout server actions
├── middleware.ts      # Route protection logic
└── app/api/auth/     # NextAuth API route (catch-all)
```

- Use JWT strategy (not database sessions) for better DX
- Store only `userId` and `role` in token
- Protect pages via middleware: check session in `middleware.ts`
- Protect API routes: check session in route handler
- Use server actions for auth mutations (login/register/logout)
- Never store raw passwords — only hashed via bcrypt (by Auth.js)

## Component Patterns

- Server components by default (fetch data directly in async components)
- Client components only when needed: interactivity, browser APIs, hooks
- Extract data fetching from components: use server actions or lib functions
- Form handling: React Hook Form + Zod validation + server actions
- Loading states: use `loading.tsx` (page-level) or `<Suspense>` (component-level)
- Error states: use `error.tsx` (page-level) or `<ErrorBoundary>` (component-level)
- Empty states: always handle arrays with `.length === 0` checks

## Server Actions

```
// src/actions/users.ts
'use server';

import { db } from '@/db';
import { users } from '@/db/schema/users';
import { eq } from 'drizzle-orm';
import { z } from 'zod';

const updateProfileSchema = z.object({
  name: z.string().min(1).max(100),
  bio: z.string().max(500).optional(),
});

export async function updateProfile(formData: FormData) {
  const session = await auth();
  if (!session?.user?.id) throw new Error('Unauthorized');

  const parsed = updateProfileSchema.parse({
    name: formData.get('name'),
    bio: formData.get('bio'),
  });

  await db.update(users)
    .set({ name: parsed.name, bio: parsed.bio, updatedAt: new Date() })
    .where(eq(users.id, session.user.id));

  revalidatePath('/profile');
}
```

Rules:
- One file per domain in `src/actions/`
- Always validate input with Zod
- Always check auth at the top
- Return typed results (never throw in production — return `{ success, error }`)
- Use `revalidatePath` or `revalidateTag` after mutations
- Never pass raw request objects — accept primitives or FormData

## Error Handling Pattern

```
// lib/errors.ts
export class AppError extends Error {
  constructor(
    message: string,
    public code: string,
    public status: number = 400
  ) {
    super(message);
  }
}

export function handleApiError(error: unknown) {
  if (error instanceof AppError) {
    return Response.json(
      { error: error.message, code: error.code },
      { status: error.status }
    );
  }
  if (error instanceof z.ZodError) {
    return Response.json(
      { error: 'Validation failed', details: error.flatten() },
      { status: 422 }
    );
  }
  console.error('Unhandled error:', error);
  return Response.json({ error: 'Internal server error' }, { status: 500 });
}
```

## Testing Conventions

- Unit tests: Vitest, co-located next to source files as `*.test.ts`
- E2E tests: Playwright in `tests/e2e/`
- DB tests: use an in-memory SQLite database (`:memory:`) with `better-sqlite3`
- Mock external services (Auth.js, email, payments) in unit tests
- Do NOT mock the database — use a test database instance

## Anti-Patterns (Never Do)

1. ❌ **`useEffect` for data fetching** — Use server components or React Query
2. ❌ **`getServerSideProps` / `getStaticProps`** — Use App Router's async components
3. ❌ **Prop drilling beyond 2 levels** — Use React context or server actions
4. ❌ **Using Prisma** — Drizzle is preferred for SQLite (lighter, better TS integration)
5. ❌ **Inline styles** — Always use Tailwind utility classes
6. ❌ **Raw SQL in components** — All DB access goes through `src/db/queries/`
7. ❌ **Any SQLite in edge runtime** — better-sqlite3 is Node-only; Turso for edge
8. ❌ **API routes for mutations** — Use server actions instead
9. ❌ **Reading `process.env` in browser code** — Use `NEXT_PUBLIC_` prefix or server-only env
10. ❌ **Outdated import paths** — Use `@/` alias, never relative imports beyond `./` or `../`

## Why These Rules (Opinionated Choices)

- **Drizzle over Prisma:** Drizzle is lighter, has zero runtime deps for SQLite, generates cleaner migrations, and maps 1:1 to SQL mental models
- **better-sqlite3 for dev:** Fast synchronous queries make dev iteration faster than async DBs
- **Server actions for mutations:** Co-locate write logic, type-safe inputs with Zod, automatic form resets
- **No pages router:** App Router is the future; mixing routers creates confusion
- **Strict TypeScript:** Catches null/undefined bugs at compile time, not runtime
